# 기술 명세서 (Tech Stack Specification)

본 기술 명세서는 **빠른 초기 개발(MVP) 시각을 완전히 배제**하고, **장기적인 서비스 운영, 견고한 유지보수성, 대규모 트래픽 대응 및 확장성**에 초점을 맞추어 단일화된 엔터프라이즈급 기술 스택을 정의합니다.

---

## 📱 1. 프론트엔드 (Frontend)

### 1.1 UI & 렌더링: React (TypeScript)
- **선정 이유**: 모바일 웹(Mobile First)으로 단일 코드베이스를 유지하여 유지보수 자원을 최소화합니다.
- **비판적 시각**: 카메라 제어 같은 하드웨어 네이티브 기능이 핵심이 아닌 이상, 모바일 앱(AOS/iOS)을 별도 개발하는 것은 기술 부채를 두 배로 늘립니다. React 웹을 고도화하여 PWA(Progressive Web App) 형태로 제공하는 것이 장기적 유지보수와 배포 관리에 훨씬 유리합니다.
- **UX 원칙**: 60대 대상 타겟을 고려한 특대형 폰트, 직관적인 고대비 색상, 최소화된 뎁스(Depth)의 화면 설계.

### 1.2 전역 상태 관리 (Client State): Zustand
- **선정 이유**: 보일러플레이트가 적어 코드가 간결하며, 특정 컴포넌트만 리렌더링하도록 세밀한 제어가 가능합니다.
- **비판적 시각**: Redux는 엄격한 아키텍처를 제공하지만 수많은 액션과 리듀서 코드로 인해 유지보수 피로도가 극심합니다. Context API는 성능 최적화가 어렵습니다. Zustand는 확장성이 뛰어나면서도 유지보수 비용이 가장 낮은 합리적인 선택입니다.

### 1.3 서버 상태 관리 (Server State): TanStack Query (React Query)
- **선정 이유**: API 요청 상태(로딩, 에러, 성공) 관리와 데이터 캐싱, 재시도(Retry) 로직을 일관되게 규격화합니다.
- **비판적 시각**: 데이터 페칭 로직을 `useEffect`로 직접 구현하면 엣지 케이스(네트워크 단절, 중복 요청 등) 대응에 구멍이 생기기 쉽습니다. 데이터 동기화와 캐싱을 프레임워크 레벨에서 통제해야 예측 가능한 프론트엔드 동작을 보장할 수 있습니다.

## ⚙️ 2. 백엔드 (Backend)

### 2.1 프레임워크: NestJS (Node.js + TypeScript)
- **선정 이유**: 의존성 주입(DI), 모듈화, 객체지향적 아키텍처(Controller-Service-Repository)를 프레임워크 차원에서 강제합니다.
- **비판적 시각**: Express는 빠르고 가볍지만, 규칙이 없어서 개발자마다 코드 스타일이 달라지고 결국 스파게티 코드로 전락합니다. NestJS는 진입 장벽이 있지만, 코드가 방대해지고 팀원이 늘어나도 일관된 구조를 유지할 수 있어 장기적인 유지보수에 필수적입니다.

### 2.2 데이터베이스 (RDBMS): PostgreSQL
- **선정 이유**: 데이터 무결성, 복잡한 조인 연산, 동시성 처리 측면에서 압도적인 안정성을 제공합니다.
- **비판적 시각**: NoSQL이나 경량 DB는 구조 변경이 쉬워 보이지만, 시세나 사용자 정보처럼 관계형 데이터가 얽혀 있는 시스템에서는 데이터 불일치(Inconsistency)라는 치명적인 문제를 낳습니다. 트랜잭션이 완벽히 보장되는 PostgreSQL을 표준으로 삼습니다.

### 2.3 ORM (DB 통신 도구): Prisma
- **선정 이유**: 완벽한 타입 추론을 제공하며, 스키마 변경 시 컴파일 단계에서 에러를 잡아줍니다.
- **비판적 시각**: TypeORM 등 기존 ORM은 데코레이터와 런타임 에러에 의존하는 경향이 있습니다. Prisma는 데이터베이스 스키마와 애플리케이션 코드를 강하게 결합하여 휴먼 에러로 인한 DB 장애를 원천 차단합니다.

### 2.4 캐시 및 세션: Redis
- **선정 이유**: 높은 처리량이 요구되는 시세 데이터 캐싱, 리프레시 토큰(세션) 관리, 동시성 제어(분산 락)를 담당합니다.
- **비판적 시각**: 모든 읽기/쓰기 부하를 메인 DB(PostgreSQL)가 감당하게 두면, 트래픽 스파이크(접속자 폭주) 시 서비스 전체가 다운됩니다. 조회 빈도가 높은 데이터는 반드시 Redis 메모리 계층에서 방어해야 합니다.

## 🚀 3. 배포 및 인프라 (DevOps / Infrastructure)

### 3.1 프론트엔드 호스팅: Vercel
- **선정 이유**: 무설정(Zero-config) 수준의 CI/CD 파이프라인과 글로벌 엣지 네트워크(CDN)를 기본 제공합니다.
- **비판적 시각**: 프론트엔드 정적 파일 서빙을 위해 AWS S3 + CloudFront를 직접 구성하고 갱신 파이프라인을 짜는 것은 인프라 전담 인력이 없는 한 불필요한 비용입니다. Vercel이 제공하는 자동화된 프리뷰 배포와 롤백 기능은 유지보수 효율을 극대화합니다.

### 3.2 백엔드 배포: Vercel Serverless Functions
- **선정 이유**: 백엔드와 프론트엔드를 동일한 Vercel 플랫폼에서 일원화 관리함으로써 인프라 구성 비용과 운영 리소스를 최소화하고, 서버리스 아키텍처를 통한 유연한 스케일링을 도모합니다.
- **비판적 시각**: 대규모 ECS 컨테이너 아키텍처는 고부하 처리에는 우수하지만 서버 상시 구동 비용과 복잡한 파이프라인 유지보수가 필요합니다. Vercel Serverless Functions는 트래픽에 맞춰 자동으로 실행 규모가 확장되고 운영 오버헤드가 없어, 린(Lean)한 개발 주기에서 높은 생산성을 제공합니다.

### 3.3 데이터베이스 호스팅: Supabase (Managed PostgreSQL)
- **선정 이유**: 복잡한 클라우드 인프라 설정 없이 클릭 몇 번으로 고성능 PostgreSQL 인스턴스와 데이터 API, 연결 풀링(PgBouncer 등)을 기본 제공받습니다.
- **비판적 시각**: AWS RDS를 다중 가용영역(Multi-AZ)으로 직접 구성하고 모니터링하는 인프라 비용은 초기 단계에서 매우 과도합니다. Supabase는 백업, 복구, 실시간 데이터 구독 기능을 갖춘 완제품 형태의 클라우드 서비스이므로 데이터 관리 비용을 획기적으로 낮춥니다.

### 3.4 크롤러 배포: Docker 컨테이너 기반 마이크로서비스
- **선정 이유**: 크롤러는 대량 데이터 조회와 장시간 대기(또는 주기적 실행)가 필요하므로 시간 제한이 있는 Vercel 서버리스에는 적합하지 않습니다. 따라서 `python:3.11-alpine` Docker 컨테이너로 빌드 후 독립된 가상 서버나 온프레미스 인프라에서 독립적으로 가동합니다.
- **비판적 시각**: 크롤러의 장애가 메인 서비스 백엔드의 성능이나 생존에 영향을 주지 않도록 마이크로서비스 형태로 격리(Fault Isolation)하여 전체 서비스의 안정성을 확보합니다.

## 🗄️ 4. 데이터베이스 및 데이터 관리 전략 (Data Management)

### 4.1 논리적 삭제 (Soft Delete)
- **선정 이유**: 사용자, 게시글, 즐겨찾기 등 주요 테이블에 `deletedAt` 컬럼을 필수로 적용합니다.
- **비판적 시각**: 사용자의 실수나 악의적인 공격으로 데이터가 삭제되었을 때 물리적 삭제(Hard Delete)를 하면 복구가 불가능합니다. 논리적 삭제를 통해 데이터 복구의 여지를 남기고 통계 데이터의 무결성을 유지해야 합니다.

### 4.2 감사 로그 (Audit Trail)
- **선정 이유**: 데이터 생성일(`createdAt`), 수정일(`updatedAt`)을 전 테이블에 강제하여 데이터 변경의 이력을 추적합니다.
- **비판적 시각**: 장애나 클레임 발생 시 데이터의 최종 변경 시점을 알지 못하면 원인 규명 자체가 불가능합니다. 이는 인프라의 블랙박스를 방지하는 최소한의 관제 장치입니다.

### 4.3 데이터 보존 및 아카이빙 (Data Retention)
- **선정 이유**: 발생 후 1년이 지난 시세 이력 등 콜드 데이터(Cold Data)는 AWS S3와 같은 저비용 스토리지로 주기적으로 이관합니다.
- **비판적 시각**: 메인 RDBMS에 과거의 무의미한 대량 데이터를 방치하면, 쿼리 속도 저하와 RDS 스토리지 비용의 기하급수적 상승을 초래합니다. 활성 데이터와 비활성 데이터를 명확히 분리해야 합니다.

### 4.4 데이터베이스 마이그레이션 (DB Migration)
- **선정 이유**: 모든 DB 스키마의 변경은 Prisma Migrate를 통해 형상 관리(버전 컨트롤)됩니다.
- **비판적 시각**: 데이터베이스 스키마를 수동으로 변경하면 로컬, 테스트, 운영 서버 간의 상태 불일치가 반드시 발생하여 배포 장애로 이어집니다. 코드 기반의 마이그레이션 도구를 통해 변경 이력을 추적하고 롤백 가능성을 확보해야 합니다.
