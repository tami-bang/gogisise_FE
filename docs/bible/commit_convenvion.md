# Git 커밋 메시지 규칙 (Commit Convention)

본 프로젝트에서는 일관된 커밋 히스토리를 유지하고 가독성을 높이기 위해 아래의 커밋 규칙을 따릅니다.

## 1. 커밋 타입 (Commit Types & Emojis)
- 🎉 `init`: 프로젝트 초기 구조 생성
- ✨ `feat`: 새로운 기능 추가 (예: 시세 조회 화면 구현, 즐겨찾기 기능 추가, 로그인 API 연동)
- 🐛 `fix`: 버그 수정 (예: 가격 계산 오류 수정, 모바일 레이아웃 깨짐 수정)
- ♻️ `refactor`: 코드 리팩토링 (예: 공통 컴포넌트 분리, API 호출 구조 개선)
- ⚡ `perf`: 성능 개선 (예: 이미지 최적화)
- 📝 `docs`: 문서 수정 (예: 기능 명세서 업데이트)
- 🔧 `chore`: 설정 변경, 의존성 추가 등 (예: Flutter 의존성 추가, Vite 설정 변경)
- 🚀 `deploy`: 배포 관련 (예: Vercel 배포)
- 🔥 `remove`: 사용하지 않는 파일 또는 컴포넌트 삭제

## 2. 커밋 원칙 (Principles)
1. **한 커밋 = 한 작업**: 한 번에 여러 개의 다른 작업을 하나의 커밋에 섞지 않습니다.
2. **제목은 50자 내외로 간결하게**: 커밋 제목만 보고도 내용을 파악할 수 있도록 작성합니다.
3. **현재형으로 작성**: 제목의 끝은 '추가', '수정', '분리', '구현', '삭제' 등으로 끝냅니다.
4. **직관적인 제목 작성**: 커밋 목록을 훑어보는 것만으로 어떤 작업이 진행되었는지 알 수 있게 합니다.

## 3. 커밋 메시지 구조 (Format)
```text
<타입>: <제목> (50자 내외로 간결하게, 현재형 작성)

상세내용
- [작업 내용 1]: 무엇을 어떻게 고쳤는지 설명
- [작업 내용 2]: 관련된 추가 설명이나 이유
```

**예시:**
```text
✨ feat: 시세 조회 화면 구현

- API(/api/v1/price)를 연동하여 현재 시세 데이터를 화면에 표시
- 로딩 상태 처리 및 스켈레톤 UI 추가
```
