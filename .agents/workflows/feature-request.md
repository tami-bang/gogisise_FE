description: 새로운 기능을 추가한다.
---

새로운 기능 추가(Feature Request) 요청이 들어왔을 때 에이전트가 수행해야 하는 표준 작업 지침이다. 임의로 독립된 가짜 HTML 스케치북 파일을 생성하여 작업 본질을 흐리지 말고, 다음 4단계 규칙을 엄격히 추종하라.

---

### 1. [명세 검증] 의존성 파일 분석
- `docs/bible/PRD.md`를 확인해서 비즈니스 타겟(60대 고령 정육점 사장님)의 가시성 및 오작동 방지 목적을 최우선 확인한다.
- 사용자가 전달한 참조문서들(docs/ 하위의 기획 및 디자인 시스템 부품/화면 명세서)을 연달아 파싱하여, 이미 정의된 변수 토큰과 UI 사양을 추출한다.

### 2. [데이터 설계] 전역 상태 및 영속성 구조화
- 기능 구현 시 화면 간 데이터 연동이 필요하다면 React의 최상위 컨텍스트(Context API 또는 전역 State)를 설계하여 데이터 흐름을 일원화한다.
- 새로고침이나 앱 재진입 시에도 유저 설정 및 데이터가 유지되도록, 상태 변화 즉시 브라우저 `localStorage`에 데이터 오브젝트를 직렬화하여 저장(setItem)하고 복원(getItem)하는 영속성 로직을 필수로 설계한다.

### 3. [코드 이관] 개발 및 배포 서버 환경 직접 반영 (`web/`)
- 모든 기능 로직과 UI 컴포넌트 코드는 현재 로컬 개발 서버(`http://localhost:5173`)가 실제로 빌드·추적하고, 향후 운영 서버로 최종 배포(Production)될 **`web/` 폴더 내의 실제 React 소스 코드 구조(`web/src/App.jsx`, `web/src/components/` 등)에 직접 반영 및 덮어쓰기하여 이관(Migration)**해야 한다.

### 4. [검증 리포트] 신뢰성 확인 및 결과 제출
- `web/` 소스 코드 수정 후 빌드 에러 유무를 확인하고, 최초 온보딩 진입 단계부터 최종 화면까지 데이터 상태가 끊김 없이 물 흐르듯 전달되는지 자가 테스트한다.
- 구현 및 이관이 완료되면 `web/` 하위에서 실제로 수정되거나 새로 생성된 파일의 상대 경로 목록과 구현 내용을 요약하여 사용자에게 단답형 리포트로 최종 전달한다.
